# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Comments App — a Next.js 16 SaaS that connects to Instagram via OAuth and runs automations: when a comment or DM matches configured keywords on a chosen post, it auto-replies and/or sends a DM (optionally generated by OpenAI). Stripe billing via Better Auth's Stripe plugin (free / pro plans), billed **per organization**. Backend uses tRPC + Prisma (PostgreSQL). UI text and domain language is Portuguese (BR) — the domain concept "organization" is spelled **Empresa** in the UI.

## Commands

Package manager is **pnpm** (Node >= 22).

- `pnpm dev` — Next dev server, bound to `127.0.0.1:3001` (not the default `0.0.0.0:3000`); React Compiler is enabled in `next.config.ts`
- `pnpm build` — runs `prisma generate && prisma migrate deploy && next build`
- `pnpm lint` — Biome check (lint + format check)
- `pnpm format` — Biome format write
- `pnpm db:migrate` — `prisma migrate dev` (create & apply a new migration locally)
- `pnpm db:generate` — regenerate Prisma client into `src/generated/prisma`
- `pnpm db:studio` — Prisma Studio
- `pnpm inngest:dev` — local Inngest dev server (required to exercise background functions; pairs with the Next dev server)

There is no test suite configured.

`prisma.config.ts` loads `DATABASE_URL` via `dotenv/config` from `.local.env` / `.env`. The Prisma client is generated to `src/generated/prisma` (not `node_modules`) — re-run `pnpm db:generate` after schema changes or imports from `@/generated/prisma/client` will be stale.

## Architecture

### Multi-tenancy model

The app is multi-tenant **by Organization** (Better Auth `organization` plugin). Domain data — automations, integrations, history, sorteios, S2S keys — is scoped to `organizationId`, not `userId`. A user can belong to multiple Empresas (roles: `owner` / `admin` / `member`) and switches between them via the sidebar's `<EmpresaSwitcher />`. The active Empresa is stored on `session.activeOrganizationId` and is what most tRPC procedures pivot on.

Key behaviors wired in `src/lib/auth.ts`:

- **User signup → default Empresa**: `databaseHooks.user.create.after` creates `Empresa de <primeiro nome>` + an `owner` Member + a default free `Subscription` row.
- **Session creation → autoseed active org**: `databaseHooks.session.create.before` picks the user's oldest membership and sets `activeOrganizationId` so the first request after login already has a tenant.
- **Plan-gated org creation**: `allowUserToCreateOrganization` permits the first Empresa always; additional ones require an active `pro` subscription on any Empresa the user owns (free = 1, pro = up to `organizationLimit: 10`, `membershipLimit: 50`).
- **Plan inheritance**: `getEffectivePlan(organizationId)` in `src/actions/subscription.ts` returns the active Empresa's own subscription if non-free, otherwise searches the owner's other Empresas for an active non-free plan. Result: upgrading one Empresa to pro automatically gives the same owner's other Empresas effective pro access. `subscription.upgrade` and `subscription.billingPortal` block re-billing when the plan is inherited and direct the user to the source Empresa.

### Routing layout (App Router)

`src/app` is split by route groups:

- `(auth)` — login / register / `forgot-password` / `reset-password`. Layout calls `auth.api.getSession` server-side and redirects authenticated users via `safeRedirect(params.redirect)` so an invitation link survives a login bounce.
- `(main)/(rest)` — authenticated app shell (dashboard, workflows, integrations, settings, help, **empresas**).
- `(main)/callback`, `(main)/workflows` — Instagram OAuth callback and automation editor.
- `(marketing)` — public landing.
- `(policies)` — legal pages sourced from `policies.md`.
- `(public)/test-inngest` — sandbox to fire Inngest events.
- `accept-invitation/[id]` — **deliberately outside `(auth)` and `(main)`** so it works for both anonymous and authenticated users. Anonymous state offers Entrar/Criar conta buttons whose hrefs include `?redirect=/accept-invitation/{id}` to preserve the invitation context across the auth roundtrip.
- `api/auth/[...all]` — Better Auth handler.
- `api/trpc/[trpc]` — tRPC handler.
- `api/webhook/instagram` — Meta/Instagram webhook (GET verify, POST events).
- `api/inngest/route.ts` — Inngest function endpoint.

### Feature-sliced backend

Domain logic lives under `src/features/<domain>/` (automations, listener, trigger, keyword, integrations, notifications, subscription, user, webhook, auth, editor, help, settings, dashboard, **organization**). Conventional shape: `server/route.ts` (tRPC router), `server/prefetch.ts` (RSC prefetch helpers), `components/`, `hooks/`, `types.ts`.

All feature routers are composed in `src/trpc/routers/_app.ts` into `appRouter`. Adding a new domain means: create `features/<x>/server/route.ts` exporting a router, then register it on `appRouter`.

### tRPC wiring

`src/trpc/init.ts` exposes three procedures:

- `baseProcedure` — public.
- `protectedProcedure` — requires a session (or an S2S API key via `ctx.s2s`); attaches `ctx.auth.{session,user}`.
- `protectedOrgProcedure` — chains on top of `protectedProcedure`. Reads `session.activeOrganizationId` (or `ctx.s2s.organizationId` for S2S), looks up the caller's `Member` row, and attaches `ctx.organizationId` + `ctx.membership` (`{ id, role, organizationId, userId }`). Throws `FORBIDDEN "Selecione uma empresa"` if no active org, or `FORBIDDEN "Sem acesso a esta empresa"` if the caller isn't a member.

Domain routers (`automations`, `integrations`, `sorteio`, `subscription`, `dashboard`, plus mutations in `keyword` / `trigger` / `listener` that reach an automation) use `protectedOrgProcedure` and filter by `ctx.organizationId`. Per-user routers (`notifications`, `user`, `editor`, `help`) stay on `protectedProcedure`.

`src/trpc/server.tsx` — server-only proxy (`trpc`), `caller` for direct server calls, `prefetch()` helper, and `<HydrateClient>` boundary for RSC → client hydration. Features expose typed prefetch helpers in `server/prefetch.ts` so server components can warm the cache before rendering client components.

`src/trpc/client.tsx` — client `<TRPCProvider>` used in `app/providers.tsx`.

### Auth & billing

`src/lib/auth.ts` configures Better Auth with the Prisma adapter, email/password + Google social, the `organization` plugin, and `@better-auth/stripe` with `organization: { enabled: true }`. The Stripe Customer is created per **Empresa** (in `organizationHooks.afterCreateOrganization`), not per User — `User.stripeCustomerId` is legacy and slated for removal in Phase 9.

`authorizeReference` gates Stripe actions: any member can `list-subscription`; `upgrade-subscription` / `cancel-subscription` / `restore-subscription` require `owner` or `admin` role on the target Empresa. The PlanTab UI mirrors this — the upgrade/manage buttons are disabled for non-owners, with an Alert explaining why.

`src/lib/auth-client.ts` registers `organizationClient()` and `stripeClient({ subscription: true })`. Use `authClient.useActiveOrganization()` / `useListOrganizations()` / `organization.{setActive, inviteMember, acceptInvitation, leave, updateMemberRole, removeMember}` on the client.

`src/lib/auth-utils.ts` holds shared helpers.

### Email (Resend)

`src/lib/email.ts` wraps the Resend SDK behind `sendEmail({ to, subject, html, text?, tag? })`. From-address defaults to `process.env.BETTER_AUTH_EMAIL` (e.g. `nasa@notifications.nasaex.com`); requires `RESEND_API_KEY`. If the key is missing (build/migrate context), `sendEmail` no-ops with a `console.warn` instead of throwing — so importing this module from server code at boot is safe.

HTML templates in the same file (PT-BR, inline styles, `escapeHtml`-guarded):

- `resetPasswordEmail` — used by `emailAndPassword.sendResetPassword`.
- `verifyEmail` — used by `emailVerification.sendVerificationEmail` (currently `sendOnSignUp: false`).
- `invitationEmail` — used by `organization.sendInvitationEmail`, links to `${APP_URL}/accept-invitation/{id}`.

Each call passes `tags: [{ name: "category", value: tag }]` so envoys can be filtered in the Resend dashboard.

### Password reset flow

1. `/login` has an "Esqueceu sua senha?" link → `/forgot-password`.
2. `ForgotPasswordForm` calls `authClient.requestPasswordReset({ email, redirectTo: "/reset-password" })`. Backend (Better Auth) issues a verification token (TTL 1h, configurable via `resetPasswordTokenExpiresIn`) and calls the `sendResetPassword` hook → Resend sends the `resetPasswordEmail` template with a link to `${BETTER_AUTH_URL}/api/auth/reset-password/{token}?callbackURL=/reset-password`.
3. User clicks the email link → Better Auth validates the token server-side → 302 to `/reset-password?token={validToken}` (or `?error=INVALID_TOKEN` if expired/consumed).
4. `ResetPasswordForm` reads the query params; on `INVALID_TOKEN` it shows an expired-link CTA, otherwise it collects a new password and calls `authClient.resetPassword({ newPassword, token })`. Success → toast + redirect to `/login`.

The client method is `requestPasswordReset` (not `forgetPassword` — that name is from older Better Auth docs).

### Webhook pipeline (Instagram)

`src/app/api/webhook/instagram/route.ts`:

1. `GET` handles Meta's `hub.challenge` verification.
2. `POST` → `parseWebhook(payload)` (`features/webhook/parser.ts`) normalizes the Meta payload into `NormalizedEvent[]` (type `MESSAGE` or `COMMENT`).
3. For each event, `ensureEventNotProcessed(eventId, type, accountId)` (`src/actions/automations`) does idempotency via the `ProcessedEvent` table — required because Meta retries.
4. `routeEvent(event)` (`features/webhook/router.ts`) dispatches to `handler/message.handler.ts` or `handler/comment.handler.ts`.

Handlers resolve the tenant by looking up `Integration.instagramId → organizationId` (the integration row is the bridge between the Meta account and the Empresa). When changing webhook behavior, preserve the idempotency check — duplicate Meta deliveries will otherwise double-send replies.

### Background jobs (Inngest)

`src/inngest/client.ts` defines the Inngest client; functions live in `src/inngest/functions.ts` and are served from `src/app/api/inngest/route.ts`. Events are dispatched via `inngest.send({ name, data })` (see `src/app/(public)/test-inngest/actions.ts` for a reference trigger). Run `pnpm inngest:dev` alongside `pnpm dev` to execute functions locally — without it, sent events accumulate but never run.

### Ecosystem sync (NASA)

The app replicates its auth entities (`User`, `Account`, `Organization`, `Member`) to a sibling app (**NASA**) so credentials are shared across the ecosystem (NASA ↔ NERP ↔ comments). This is **distinct** from the per-org S2S consent flow (`x-comments-*` headers over `CommentsIntegrationKey`) — sync is the master app↔app channel keyed on `SYNC_SHARED_SECRET` + `SYNC_API_KEY`, identical across all apps.

- **Signing — `src/lib/ecosystem-sync/cred.ts`**: the canonical string is `` `${METHOD}\n${pathname}\n${body}\n${timestamp}` `` (pathname **without query**), signed `HMAC-SHA256` → hex over `SYNC_SHARED_SECRET`, carried in `x-sync-api-key` / `x-sync-timestamp` / `x-sync-signature`. **This must match byte-for-byte across every app in the ecosystem — never change the canonical format or header names.** `buildSyncHeaders` signs OUTBOUND; `verifyEcosystemSyncRequest` verifies INBOUND (timing-safe, ±5min drift window, never throws → 401 on any failure).
- **Outbound client — `src/lib/ecosystem-sync/nasa.ts`**: `ecosystemSyncNasa.upsert{User,Account,Org,Member}` POST to NASA's `/api/sync/comments` (base from `NASA_SYNC_BASE_URL` ?? `NASA_BASE_URL`). Non-2xx throws so the Inngest step retries.
- **Payloads — `src/lib/ecosystem-sync/payloads.ts`**: canonical shapes mirrored from NASA. The cuid `id` is always propagated — it's the upsert idempotency key. Dates travel as ISO strings; each side reads only the fields its schema has (comments has no `phone`; sent as `null`).
- **Outbound jobs — `src/inngest/functions/sync/`**: one function per entity (`replicate-*-to-nasa.ts`), 5 retries each. `replicate-member-to-nasa.ts` is **self-sufficient** — it upserts the whole graph in FK order (user → accounts → org → member), so any route that creates a member only needs to emit `memberId`.
- **Inbound endpoint — `src/app/api/sync/nasa/route.ts`** (NASA → comments): verifies the signature, then **uses raw Prisma only — NEVER `auth.api.*`** (that would fire `databaseHooks.user.create.after` and spawn a phantom Empresa + duplicate Member/Subscription). All upserts are by `id` (idempotent). Missing-FK prerequisites return `409 { retryable: true }` so NASA re-enqueues until they converge; unique collisions (email, org slug, member pair) are logged and skipped — except org slug, which writes `null` rather than skipping the whole org (skipping would 409 its members forever).
- **Emit points — `src/lib/auth.ts`**: hooks `databaseHooks.user.create.after` (emits user + default org + member), `databaseHooks.account.create.after`, and `organizationHooks.{afterCreateOrganization, afterAddMember, afterAcceptInvitation}`. All wrapped in `try/catch` that only logs — sync is **best-effort and must never break sign-up/org creation**. Note `afterAcceptInvitation` is required separately: better-auth does **not** fire `afterAddMember` when a member joins via invitation.

Required env: `SYNC_SHARED_SECRET`, `SYNC_API_KEY`, `NASA_SYNC_BASE_URL` (or `NASA_BASE_URL`); optional `SYNC_REQUEST_TIMEOUT_MS` (default 10s). New sync functions must be registered in `src/app/api/inngest/route.ts`.

### Database

`prisma/schema.prisma`. Key relations:

- `Organization → { Member[], Invitation[], Automation[], Integration[], Sorteio[], History[], CommentsIntegrationConsent[], CommentsIntegrationKey[] }`. Organizations carry their own `stripeCustomerId`.
- `Automation → { Listerner (1:1), Trigger[], Keyword[], Post[], Dms[] }`. **`Listerner` is the table name — keep the typo**; it holds the AI prompt, comment reply template, and counters (`dmCount`, `commentCount`).
- `User → { Member[], Sessions, Accounts }` — Better Auth core. `User.stripeCustomerId` is legacy (free-tier remnant); Empresa is the Stripe customer now.
- `Subscription` (Better Auth Stripe plugin) — `referenceId = Organization.id`, not `User.id`. Not connected by FK.
- `Notification` and `NotificationPreference` stay per-user (`NotificationPreference` is 1:1).
- `ProcessedEvent` — webhook idempotency table.

**Phase 9 is intentionally deferred**: `Automation`, `Integration`, `Sorteio`, `History`, `CommentsIntegrationConsent`, `CommentsIntegrationKey` still carry a nullable legacy `userId` column. Code reads only `organizationId`; the column drop + NOT NULL tightening is a future migration. Don't reintroduce `userId` reads.

Enums: `INTEGRATIONS` (only `INSTAGRAM` today), `MEDIA_TYPE` (`IMAGE | VIDEO | CAROUSEL_ALBUM`), `LISTENERS` (`SMARTAI | MESSAGE` — SMARTAI = OpenAI-generated reply, MESSAGE = static template), `NOTIFICATION_TYPE`, `Plan` (`free | pro`).

### UI

shadcn/ui (`components.json`) with Tailwind v4 and `tw-animate-css`; primitives in `src/components/ui`. Theme via `next-themes`. Forms use `react-hook-form` + `zod` resolvers. Data fetching is tRPC + TanStack Query; URL state via `nuqs`.

Auth forms (login, register, accept-invitation) use a `safeRedirect(value)` helper that rejects non-`/`-relative or `//`-prefixed paths to block open-redirect attacks. The `?redirect=` query param is preserved end-to-end so invitation links survive the login/register roundtrip.

## Conventions

- Biome (not ESLint/Prettier) with 2-space indent. `next` and `react` linter domains are on; `suspicious/noUnknownAtRules` is off (Tailwind). Biome auto-sorts imports — third-party first, `@/*` aliased imports after.
- Path alias `@/*` → `src/*` (`tsconfig.json`).
- React Compiler is enabled (`next.config.ts` + `babel-plugin-react-compiler`) — avoid manual `useMemo`/`useCallback` unless needed.
- The Prisma client is imported from `@/generated/prisma/client`, not `@prisma/client`. Use the singleton from `@/lib/db` (handles dev hot-reload + uses `PrismaPg` adapter).
- For async form data (RHF), prefer the `values` prop over `defaultValues + useEffect + form.reset` — it's the canonical pattern for data that arrives after first render.
- Domain mutations that reference an `automationId` directly (`keyword`, `trigger`, `listener`) must validate `automation.organizationId === ctx.organizationId` before writing — they don't carry the org column themselves.
- `next.config.ts` has an `allowedDevOrigins` entry for an ngrok URL — that's how the Instagram webhook is exposed during local dev.
